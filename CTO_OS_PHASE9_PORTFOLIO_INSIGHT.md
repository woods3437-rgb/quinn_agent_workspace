# Phase 9 Portfolio + Insight Layer

Phase 9 turns the CTO OS into a system-level control room: a cross-project
control room, system-wide shipped + risk + decision views, reusable
playbooks distilled from completed work, outcome scoring, and an opt-in
notification boundary.

## New Modules

| Module | Purpose |
|--------|---------|
| `cto_os_api/control_room.py` | One-glance system status aggregate. |
| `cto_os_api/system_shipped.py` | Cross-project "what we shipped" + velocity windows. |
| `cto_os_api/risk_concentration.py` | Per-project risk severity rollup, recurring themes, no-mitigation flags. |
| `cto_os_api/decision_graph.py` | Project + system decision graphs (nodes + edges). |
| `cto_os_api/playbook_service.py` | Generate playbooks from completed sessions (secrets-sanitiser) and apply to tasks. |
| `cto_os_api/outcome_scoring.py` | Record + roll up scores; one-line hint for prompts. |
| `cto_os_api/notifications.py` | Three-gate notification service (off by default). |
| `cto_os_api/staleness.py` | Detects 14d-inactive projects, 7d-stale high-priority tasks, etc. |

## System-level safety model

- **No cross-project memory bleed.** Every aggregator iterates per project and never calls `memory_engine.search(cross_project=True)`. Aggregates expose metadata (counts, titles, IDs), not memory content.
- **Decision graphs** show titles and IDs across projects; navigating into a project preserves project-scoped retrieval. Each surface caps to 50 decisions/project.
- **Playbooks sanitise on extraction.** Any step / risk / acceptance line containing a secret-shaped token (`password|secret|api[_-]?key|token`) or `.env*` is dropped before the playbook is stored. `source_project_id` is recorded for audit.
- **Outcome scores** clamp to 1â€“5 server-side; only one-line summaries feed back into LLM prompts as soft hints.

## Notification safety model (three gates)

1. `CTO_OS_ENABLE_NOTIFICATIONS=1` in env (process kill-switch)
2. `rule.enabled == true`
3. `destination` validates per channel â€” HTTPS-only for webhook/slack/discord; email regex for `email` (SMTP delivery itself is **not** implemented in Phase 9 â€” the channel is recognised but unwired).

Failure on any gate writes a `NotificationEvent(status=skipped)` with the gate reason. No outbound HTTP. The test endpoint follows the same gates. Secrets are referenced by `secret_ref` name; the rule never stores the value.

## Recommended Phase 10

Phase 9 makes the system observable across projects. Phase 10 should make
the assistant **proactive** without breaking safety:

1. **Daily / weekly digest** generated by an internal scheduler, posted to a configured notification rule. Disabled until both env flag AND rule are enabled.
2. **Signed webhook deliveries** â€” add HMAC signing for outbound notifications, with rotation. Couples with `secret_ref`.
3. **Embeddings-based playbook matching** â€” when a new task is created, suggest a playbook based on title/description similarity (project-local only by default; cross-project opt-in).
4. **Outcome-score-aware prompts** â€” feed `OutcomeScoringService.prompt_hint(project_id)` into agent system prompts so the model is calibrated by past accuracy.
5. **Read-only public status page** behind the internal auth token â€” useful for sharing a status URL with a small invited group without exposing project memory content.
